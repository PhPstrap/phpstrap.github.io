🔐 Sign in with Google — Module Setup

This guide walks through creating a Google OAuth 2.0 application, running the database migration, enabling the module, and integrating the sign-in button with your PhPstrap login page.

TL;DR: Create Google credentials → run migrate.php → enable the module in Admin → Modules → paste your Client ID & Secret → the button appears on /login/ automatically.

1) How It Works

The module implements the OAuth 2.0 Authorization Code flow using PHP's built-in cURL — no Composer packages required.

  1. User clicks Continue with Google on the login page.
  2. They are redirected to Google to choose an account and grant access.
  3. Google redirects back to /modules/google-oauth/callback.php with a one-time code.
  4. The module exchanges the code for an access token, then fetches the user's profile.
  5. The user is matched to a PhPstrap account by the following priority:
    • Existing account with the same google_id (returning Google user).
    • Existing account with the same email address (links Google to the account on first use).
    • No match → new account created automatically (if registration is open).
  6. A standard PhPstrap session is started and the user lands on the dashboard.

2) Module Files

The login page (login/index.php) is patched to auto-load the button when the module is enabled and configured.

3) Prerequisites

  • PHP 7.4+ with the cURL and JSON extensions enabled.
  • Your site must be accessible over HTTPS in production (Google requires it for the redirect URI).
  • A Google account with access to Google Cloud Console.
  • The PhPstrap modules table must exist (created during install).

4) Create Google OAuth Credentials

Step 1 — Create or select a project

  1. Go to console.cloud.google.com.
  2. Click the project dropdown at the top → New Project.
  3. Give it a name (e.g., My PhPstrap App) and click Create.

Step 2 — Configure the OAuth consent screen

  1. In the left sidebar go to APIs & Services → OAuth consent screen.
  2. Choose External (unless this is a Google Workspace app for a single org).
  3. Fill in the required fields:
    • App name — shown to users on the consent screen (e.g., My Site).
    • User support email — your email address.
    • Developer contact information — your email address.
  4. Under Scopes, click Add or Remove Scopes and add:
    • .../auth/userinfo.email
    • .../auth/userinfo.profile
    • openid
  5. Save and continue through the remaining steps.
Publishing status: While in Testing mode only accounts you add as test users can sign in. Click Publish App once you are ready to allow all Google accounts.

Step 3 — Create credentials

  1. Go to APIs & Services → Credentials.
  2. Click + Create Credentials → OAuth client ID.
  3. Set Application type to Web application.
  4. Under Authorized redirect URIs, add: 
    https://yourdomain.com/modules/google-oauth/callback.php
    Replace yourdomain.com with your actual domain. For local development you can also add: 
    http://localhost/modules/google-oauth/callback.php
  5. Click Create.
  6. Copy your Client ID and Client Secret — you will need them in step 6.

5) Run the Database Migration

The migration adds a google_id column to the users table. It is safe to run multiple times.

Option A — Via browser (recommended)

Log in as an admin, then visit:

https://yourdomain.com/modules/google-oauth/migrate.php

You should see: Migration complete: google_id column added to users table.

Option B — Via CLI

php /path/to/your/site/modules/google-oauth/migrate.php
What it does: Runs ALTER TABLE users ADD COLUMN google_id VARCHAR(100) NULL DEFAULT NULL and adds an index. If the column already exists, it prints a notice and exits cleanly.

6) Enable the Module & Enter Credentials

  1. Log in to the PhPstrap admin panel.
  2. Navigate to Admin → Modules.
  3. Click Scan for Modules — Sign in with Google should appear.
  4. Click Enable.
  5. Open the module's Settings and fill in:
    • Client ID — from Google Cloud Console.
    • Client Secret — from Google Cloud Console.
    • Allowed Domains — optional; comma-separated list of email domains that are permitted to sign in (e.g., company.com, partner.org). Leave blank to allow all Google accounts.
  6. Save settings.

The Continue with Google button will now appear automatically on /login/.

7) Login Page Integration

The login/index.php file was patched during module installation to auto-load the Google button. No additional changes are required.

If you need to add the button to a custom login page, insert this snippet where you want the button to appear:

<?php
// Check module status and render button
$stmt = $pdo->prepare("SELECT enabled, settings FROM modules WHERE name = 'google-oauth'");
$stmt->execute();
$gm = $stmt->fetch();
if ($gm && $gm['enabled']) {
    $gs = json_decode($gm['settings'], true);
    if (!empty($gs['client_id']) && !empty($gs['client_secret'])) {
        require_once '../modules/google-oauth/GoogleOAuth.php';
        $oauth = new PhPstrap\Modules\GoogleOAuth\GoogleOAuth();
        $oauth->init();
        echo '<div class="google-oauth-divider">or</div>';
        echo $oauth->renderButton('Continue with Google');
    }
}
?>

Also add the stylesheet in the <head>:

<link href="/modules/google-oauth/assets/google-oauth.css" rel="stylesheet">

8) Settings Reference

SettingRequiredDescription
client_id Yes OAuth 2.0 Client ID from Google Cloud Console.
client_secret Yes OAuth 2.0 Client Secret from Google Cloud Console. Stored in the database — never commit to version control.
allowed_domains No Comma-separated email domains allowed to sign in (e.g., acme.com, partner.org). Leave blank to allow any Google account.

9) User Account Behaviour

ScenarioResult
Returning user who previously signed in with Google Matched by google_id — logged in immediately.
Existing PhPstrap account with the same email Google ID is linked to the existing account on first use; the user is logged in.
New user, registration open Account created automatically with the name and email from Google. verified = 1 is set (no email verification needed).
New user, registration closed (registration_enabled = 0) Sign-in is rejected with an error message.
Email domain not in allowed_domains Sign-in is rejected with an error message.
Password: Users created via Google sign-in have no password set. They can still set one later via the password reset flow if you want to allow both login methods.

10) Security Notes

  • State parameter: A random CSRF token is generated for every auth request and verified on callback, preventing cross-site request forgery.
  • HTTPS only: Always use HTTPS for the redirect URI in production. Google will refuse plaintext HTTP callbacks (except localhost).
  • SSL verification: All cURL requests to Google APIs use CURLOPT_SSL_VERIFYPEER = true.
  • Domain allow-listing: Use allowed_domains when deploying for an organisation to prevent arbitrary Google accounts from registering.
  • Client Secret: Stored in the modules database table, not in source files. Restrict database access accordingly.

11) Local Development

  1. Add http://localhost/modules/google-oauth/callback.php (or your local equivalent) as an authorised redirect URI in Google Cloud Console.
  2. Add your Google account as a test user on the OAuth consent screen while the app is unpublished.
  3. Ensure BASE_URL in config/app.php points to your local URL so the callback URI is built correctly.
Note: Google may show an "App not verified" warning during testing. Click Advanced → Go to <app name> (unsafe) to proceed. This warning disappears once the app is published.

12) Troubleshooting

  • Button does not appear on login page
    • Confirm the module is enabled in Admin → Modules.
    • Confirm both client_id and client_secret are saved in module settings.
    • Check PHP error logs for an initialization error.
  • redirect_uri_mismatch error from Google
    • The redirect URI in Google Cloud Console must exactly match BASE_URL . '/modules/google-oauth/callback.php'. Check for trailing slashes or http vs https mismatches.
  • Migration fails with a permissions error
    • The database user must have the ALTER privilege on the users table. Grant it or run the migration manually:
    ALTER TABLE users
  ADD COLUMN google_id VARCHAR(100) NULL DEFAULT NULL AFTER email,
  ADD INDEX idx_google_id (google_id);
  • State mismatch error on callback
    • Sessions must be running before the auth URL is generated. Verify initializeApp() is called in login/index.php before the button renders.
    • Also check that your session cookie is not blocked by a browser extension or a misconfigured SameSite attribute.
  • cURL error / "Failed to exchange authorization code"
    • Verify outbound HTTPS traffic to oauth2.googleapis.com is allowed by your server's firewall.
    • Confirm the PHP cURL extension is installed: php -m | grep curl.
  • "Sign-in rejected — domain not permitted"
    • The user's email domain is not in the allowed_domains list. Either add the domain or clear the field to allow all domains.
  • User logged in but profile has no name
    • Google's name field is empty for some accounts. The module falls back to the email prefix. The user can update their name in Profile settings.
